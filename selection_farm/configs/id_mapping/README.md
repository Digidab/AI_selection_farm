# Система генерации и маппинга ID (ID Mapping System)

## Контекст и назначение
Данная директория содержит правила, справочники и логику для генерации уникальных идентификаторов (ID) в рамках проекта `AI_selection_farm`. ID используются для сквозного отслеживания процессов, моделей, эпох, поколений, родословных и других сущностей фермы искусственного отбора.

**Критически важно для AI-агентов:** При создании новых сущностей или автоматической генерации ID строго следуйте описанной ниже спецификации. Любые изменения в маппинге префиксов должны фиксироваться в файле `id_directory.json`.

---

## Спецификация формата ID

По умолчанию ID состоит из **8 символов** и имеет регулярную структуру: `[Префикс(2)][Индекс(2)][Счетчик(4)]` (например, `MO000001`).

### 1. Блок природы сущности (Символы 1–2)
Первые два символа — это латинские буквы в верхнем регистре (**UPPERCASE**), отражающие "природу" или тип абстракции.
* `PR` — Процессы (Processes)
* `MO` — Модели (Models)
* `EP` — Эпохи (Epochs)
* `GE` — Поколения (Generations)
* `PE` — Родословные (Pedigree)
* `RU` — Запуски (Runs) — `run_id`
* `DA` — Датасеты (Datasets) — `dataset_id`
* `TC` — Конфигурации обучения (Training Configs) — `training_config_id`
* `SV` — Версии селектора (Selector Versions) — `selector_version`
* `BR` — Отчёты Bereiter (Bereiter Reports) — `bereiter_report_id`

Полный и всегда актуальный список — в `id_directory.json`; список выше поддерживается синхронно вручную для быстрого чтения человеком.

### 2. Блок порядкового номера абстракции (Символы 3–4)
Следующие 2 символа — это цифры (от `00` до `99`), определяющие уникальный индекс конкретной абстракции. 
* Используется для предотвращения коллизий, если названия разных абстракций начинаются на одни и те же буквы.
* *Пример:* Для сущности "Модели" зарезервирован префикс с индексом `MO00`. Если в систему вводится новая абстракция "Мощность", которая также претендует на `MO`, ей неизменно присваивается индекс `MO01`.

### 3. Блок порядкового счетчика (Символы 5–8+)
Оставшиеся символы — это возрастающий цифровой счетчик конкретных экземпляров сущности, начиная с `0000`, затем `0001`, `0002` и так далее.

---

## Правила масштабирования и переполнения (Overflow Rules)

Система является динамически расширяемой в сторону увеличения разрядности счетчика при достижении лимитов:

1.  **Базовый лимит (4 знака):** Диапазон счетчика от `0000` до `9999` (всего 10,000 ID). Суммарная длина ID = 8 символов.
2.  **Первое расширение (5 знаков):** Когда счетчик достигает предела (например, `MO009999`), к цифровому блоку счетчика **добавляется 5-й знак**. Следующий ID будет выглядеть как `MO0010000`. Длина ID становится равной 9 символам.
3.  **Второе расширение (6 знаков):** Когда заканчивается пятизначный блок (`MO0099999`), счетчик расширяется до 6 знаков. Следующий ID: `MO00100000`. Длина ID становится равной 10 символам.
4.  Этот же принцип (+1 символ к длине счетчика) применяется при любом последующем исчерпании текущего числового блока.

## Файлы в этой директории

| Файл | Роль |
|---|---|
| `id_directory.json` | Справочник значений префиксов (какие буквы что означают). Меняется только при добавлении нового типа сущности, никогда — при обычной выдаче ID. |
| `id_counters.json` | Фактическое состояние счётчиков: последний выданный номер по каждому `full_prefix`. Меняется при **каждой** выдаче нового ID. |

Разделение принципиальное: `id_directory.json` — это словарь ("что значит MO00"), `id_counters.json` — это состояние ("на чём мы остановились для MO00"). Не путайте и не смешивайте их.

### Структура `id_counters.json`

```json
{
  "MO00": {
    "last_issued": 42,
    "counter_width": 4,
    "last_full_id": "MO000042",
    "updated_at": "2026-07-06T16:00:00"
  }
}
```

* `last_issued` — последнее выданное числовое значение счётчика (`null`, если для этого префикса ещё не выдано ни одного ID).
* `counter_width` — текущая разрядность цифрового блока счётчика (растёт по правилам переполнения: 4 → 5 → 6 → ...).
* `last_full_id` — последний полностью собранный ID, для быстрой сверки без пересчёта.
* `updated_at` — время последнего выпуска ID для этого префикса.

Следующий ID = `last_issued + 1` (или `0`, если `last_issued` равен `null`), отформатированный с учётом `counter_width`. Если новое значение больше не помещается в текущую разрядность (например, `9999 + 1 = 10000` не влезает в `counter_width: 4`), `counter_width` увеличивается на 1 **перед** форматированием, согласно правилам переполнения выше.

### Атомарность — обязательное требование

`id_counters.json` — это разделяемое состояние. Если ID могут запрашивать несколько процессов параллельно (selector, bereiter, trainer), чтение-инкремент-запись должны быть атомарными:

* при файловой реализации — использовать блокировку файла (например, `fcntl.flock` в Python) на время чтения+записи;
* либо, что надёжнее при конкурентном доступе, перенести состояние счётчиков в PostgreSQL (`db/`) и использовать `SEQUENCE`/`SERIAL` на каждый `full_prefix` вместо JSON-файла.

## Генератор ID: `scripts/generate_id.py`

Логика выше реализована и протестирована (в т.ч. 30 параллельных запросов без дублей — блокировка через `fcntl.flock` работает).

CLI:

```bash
python3 scripts/generate_id.py models      # по английскому имени сущности -> MO000000
python3 scripts/generate_id.py "Модели"    # по русскому имени -> тот же MO00
python3 scripts/generate_id.py MO00        # напрямую по full_prefix
```

Python API:

```python
from scripts.generate_id import generate_id, parse_id

model_id = generate_id("models")     # "MO000000"
parts = parse_id(model_id)           # {"prefix": "MO", "index": "00", "counter": "000000"}
```

Важно для реализации/парсинга: счётчик растёт в длину со временем, поэтому разбор ID всегда должен быть `prefix=id[:2]`, `index=id[2:4]`, `counter=id[4:]` — никогда не `id[4:8]` с фиксированной длиной, иначе парсинг сломается после первого переполнения. `parse_id()` в `generate_id.py` уже реализует это корректно — используйте её вместо ручного слайсинга в других модулях.

Если запрошенная сущность не зарегистрирована в `id_directory.json`, `resolve_full_prefix()` **выбрасывает `UnknownEntityError`** вместо того, чтобы молча создать новый префикс — это намеренно, см. правило ниже.

## Инструкция по применению для AI-агентов
1.  Перед присвоением ID определите тип сущности.
2.  Сверьтесь с файлом `id_directory.json` (или вызовите `generate_id.py` — он сделает это сам), чтобы узнать текущий буквенно-цифровой префикс (например, `GE00`).
3.  Если сущности нет в справочнике, **предложите** новую запись пользователю на подтверждение (проверив пересечения по буквам: при совпадении букв инкрементируйте индекс `00` -> `01`). Не создавайте новый префикс самостоятельно без подтверждения — присвоенный индекс становится неизменным. `generate_id.py` уже отказывает в выдаче ID для незарегистрированной сущности.
4.  Для выдачи ID используйте `generate_id()` из `scripts/generate_id.py` — она атомарно читает `last_issued`/`counter_width` из `id_counters.json`, применяет правила переполнения, записывает обновлённое состояние и возвращает новый ID. Не реализовывайте эту логику заново вручную в других местах проекта.
