# ID Domains — что нужно сейчас, что откладываем, что не нужно вовсе

## Контекст

Этот файл фиксирует решение по каждому ID-домену: активен ли он прямо сейчас в
`scripts/id_generator/core.py`, отложен до появления соответствующей подсистемы, или вовсе не должен
проходить через нашу ID-систему. Решение принимается один раз и меняется явно — префикс,
получивший `status: "active"`, не переименовывается и не удаляется.

Источники истины для этого решения: `readmy_info/selection_farm_database_guide.md` (§3, §8) —
конкретная схема v001 с явным списком того, что "не заслуживает своей сложности" на текущем
этапе, и `readmy_info/PROJECT_PHILOSOPHY.md` (§9) — обязательные поля паспорта модели.

## Активно сейчас (v001 core — реальная таблица в БД уже спроектирована)

| Префикс | Домен | Источник обязательности |
|---|---|---|
| `MO` | Модели (`model_id`) | `farm.model_registry`, паспорт модели |
| `RU` | Запуски (`run_id`) | `farm.runs` — одна строка на исполнение Selector/Bereiter/Trainer |
| `TA` | Задачи (`task_id`) | `farm.tasks` — резюмируемая очередь работы |
| `GN` | Вывод модели (`generation_id`) | `farm.generations` — сырой ответ модели на задачу |
| `VA` | Результаты валидации (`validation_id`) | `farm.validation_results` — вердикт Selector'а |
| `SA` | Сэмплы (`sample_id`) | `farm.samples` — итоговый accepted/rejected сэмпл |
| `EM` | Эмбеддинги (`embedding_id`) | `farm.embeddings` — векторы для pgvector-дедупликации |

Все семь — это ровно семь таблиц из database_guide.md §3 ("v001 core — block scheme"). Ничего
не добавлено сверху и ничего не выброшено — набор доменов один в один совпадает с тем, что
реально нужно, чтобы прогнать task → generation → validation → sample от начала до конца.

## Отложено (префикс зарезервирован, генерация ID заблокирована)

| Префикс | Домен | Условие активации |
|---|---|---|
| `GE` | Поколения разведения (breeding generation number) | Нужно ≥2 поколения моделей и хотя бы одно реальное решение о разведении — сейчас моделей нет вообще (database_guide.md §8) |
| `PE` | Родословные (`farm.model_lineage`) | Явно отложена в §8: нужна минимум вторая модель, чтобы связь родитель/потомок вообще существовала. До этого момента используется `studbook/model_lineage/lineage.yaml` |
| `BR` | Отчёты Bereiter | Bereiter (`services/bereiter`) ещё не реализован; `farm.bereiter_trials`/`bereiter_trial_results` отложены в §8 |
| `EP` | Эпохи обучения | Trainer (`services/trainer`) ещё не реализован; `farm.training_jobs` отложена в §8 |

`scripts/id_generator/core.py` физически откажет в выдаче ID для этих префиксов (`DeferredEntityError`),
пока их `status` в `id_directory.json` не будет вручную переведён в `"active"` — и это должно
происходить только вместе с реализацией соответствующей таблицы/сервиса, не раньше.

## Никогда не проходят через ID-generator (не домены нашей ID-системы)

| Что | Почему не ID-домен |
|---|---|
| `dataset_id` | Текстовая метка, привязанная к имени файла версии (`golden_dataset_v001.jsonl`), не FK ни на какую таблицу — database_guide.md §9.2 и §5 (rule 5) прямо запрещают заводить под это отдельную сущность раньше времени |
| `training_config_id` | Ссылается на файл в `configs/trainer/` (например `lora_train_v001.yaml`), не на таблицу |
| `selector_version` | Ссылается на файл в `configs/selector/` (например `selector_v001.yaml`), не на таблицу |
| Процессы (`PR`, было в старой версии справочника) | Дублирует `RU`/`farm.runs`, которая уже покрывает "одно исполнение Selector/Bereiter/Trainer" — самостоятельного смысла не несёт, удалено из справочника |

## Важное разграничение: GE vs GN

Слово "поколение" (generation) в проекте означает две разные вещи, и это специально разведено
на два префикса, чтобы не перепутать их в коде или в данных:

```text
GE = поколение в смысле разведения (breeding generation)
     — studbook/generation_reports/generation_NNN/, farm.model_registry.generation (int)
     — сущность верхнего уровня: "N-й цикл отбора моделей"

GN = один сырой вывод модели (model generation / raw output)
     — farm.generations.generation_id
     — сущность нижнего уровня: "модель ответила на одну конкретную задачу"
```

Одна модель за один запуск (`run`) генерирует множество `GN`-записей (по одной на задачу).
Одно поколение разведения (`GE`) объединяет множество моделей и их `GN`-записи в рамках цикла
отбора. Не используйте один префикс для обоих смыслов.

## Правило на будущее

Новый домен получает `status: "active"` только тогда, когда для него уже существует (или
одновременно создаётся) конкретный потребитель: таблица в БД, обязательное поле в уже принятом
документе (`PROJECT_PHILOSOPHY.md`, `database_guide.md`), или файл, который реально пишется
кодом. "Может пригодиться" — недостаточное основание; см. пример с `PR`, который был добавлен
по одной лишь аналогии и оказался дублем `RU`.
