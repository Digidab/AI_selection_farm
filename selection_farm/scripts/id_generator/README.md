# id_generator — миссия директории и полная карта контекста по ID

Если ты агент и тебе нужно создать, разобрать (parse) или зарегистрировать ID в проекте
`AI_selection_farm` — прочитай этот файл целиком **до** того, как писать код. Он не повторяет
остальные документы по ID, а связывает их в одну картину и говорит, что в каком файле искать.

---

## 1. Миссия этой директории

`scripts/id_generator/` — единственное место в проекте, где живёт код, порождающий ID. Ничто
за пределами этой директории не должно самостоятельно реализовывать генерацию, парсинг или
инкремент счётчика ID — только импортировать отсюда.

Директория содержит два слоя:

```text
core.py         — движок: атомарная генерация, overflow, парсинг, разрешение префикса
<entity>_id.py  — по одной обёртке на каждый АКТИВНЫЙ ID-домен, файл назван по сущности,
                  функция внутри — issue_<entity>_id()
```

### Зачем нужен слой обёрток поверх core.py

`core.py` (алгоритм генерации) уже универсален и принимает произвольный ключ — но именно поэтому
вызывающий код (`services/selector`, `services/bereiter`, `services/trainer`, studbook-writer'ы)
рисковал бы разбрасывать по себе строки вида `generate_id("models")`, `generate_id("model")`,
`generate_id("Models")` — опечатка не поймается на этапе написания кода, а обнаружится только в
рантайме через `UnknownEntityError`.

Обёртки `issue_model_id()`, `issue_run_id()` и так далее решают это: каждая жёстко зашита на свой
`full_prefix`, импортируется по имени (автодополнение в IDE, никаких строк), и это единственный
санкционированный способ запросить ID для соответствующей области. Если в будущем что-то в
`selector`/`bereiter`/`trainer` вызывает `generate_id()` напрямую со свободной строкой — это
сигнал, что для этой области забыли завести обёртку здесь.

### Таблица активных доменов → обёртка → префикс

Файл называется по сущности, функция внутри — `issue_<entity>_id()`:

| Домен | Файл | Функция | full_prefix |
|---|---|---|---|
| Модели (`model_id`) | `model_id.py` | `issue_model_id()` | `MO00` |
| Запуски (`run_id`) | `run_id.py` | `issue_run_id()` | `RU00` |
| Задачи (`task_id`) | `task_id.py` | `issue_task_id()` | `TA00` |
| Вывод модели (`generation_id`) | `generation_id.py` | `issue_generation_id()` | `GN00` |
| Результаты валидации (`validation_id`) | `validation_id.py` | `issue_validation_id()` | `VA00` |
| Сэмплы (`sample_id`) | `sample_id.py` | `issue_sample_id()` | `SA00` |
| Эмбеддинги (`embedding_id`) | `embedding_id.py` | `issue_embedding_id()` | `EM00` |

Отложенные домены (`GE`, `PE`, `BR`, `EP`) обёрток пока не получают — для них `core.py` и так
физически откажет через `DeferredEntityError`. Обёртка появляется только вместе с активацией
домена (см. `ID_DOMAINS.md`, пункт 4 ниже).

Помимо обёрток, здесь же со временем будут жить прочие автоматизации, завязанные на ID —
например аудит целостности `id_counters.json`, безопасная активация отложенного домена, массовая
сверка ID в `studbook/` против счётчиков и т.п.

## 2. Полная карта контекста по ID — читать в этом порядке

Это квинтэссенция: если прочитать только этот раздел, контекст по ID восстанавливается полностью
без необходимости искать остальные файлы вручную.

1. **`configs/id_mapping/README.md`** — спецификация формата ID:
   - структура `[Префикс(2)][Индекс(2)][Счётчик(4+)]`, например `MO000000`;
   - индекс (символы 3–4) разруливает коллизии, если два разных домена метят на одни и те же
     буквы (пример в файле: гипотетическая "Мощность" получила бы `MO01`, если бы претендовала
     на `MO`);
   - правила переполнения счётчика: 4 знака (`0000`–`9999`) → 5 знаков (`10000`–`99999`) → 6 и
     так далее, без потери и дублирования значений;
   - требование атомарности (`fcntl.flock` на `id_counters.json`) — потому что
     selector/bereiter/trainer могут запрашивать ID параллельно;
   - инструкция для агентов: не изобретать новый префикс самостоятельно, предлагать пользователю
     на подтверждение.

2. **`configs/id_mapping/id_directory.json`** — живой словарь префиксов: какая буквенная пара что
   означает, и поле `status` (`active` / `deferred`) с `activation_condition` для отложенных.
   Меняется только при добавлении/активации нового домена — никогда при обычной выдаче ID.

3. **`configs/id_mapping/id_counters.json`** — фактическое состояние: последний выданный номер на
   каждый `full_prefix`. Меняется при каждой выдаче. Трогать вручную нельзя — только через
   `core.py`, который держит файловую блокировку на время чтения-инкремента-записи.

4. **`configs/id_mapping/ID_DOMAINS.md`** — обоснование "почему": для каждого домена — активен он
   сейчас, отложен, или вообще не должен проходить через эту ID-систему, со ссылками на
   `readmy_info/selection_farm_database_guide.md` (§3 — v001 core из семи таблиц, §8 — что
   явно отложено) и `readmy_info/PROJECT_PHILOSOPHY.md` (§9 — обязательные поля паспорта модели).
   Здесь же явно разведены `GE` (поколение в смысле разведения) и `GN` (сырой вывод модели) —
   два разных понятия, случайно обозначаемых одним английским словом "generation".

5. **`scripts/id_generator/core.py`** (эта директория) — сама реализация: `generate_id()`,
   `resolve_full_prefix()`, `parse_id()`, исключения `UnknownEntityError` (домен не
   зарегистрирован) и `DeferredEntityError` (домен зарегистрирован, но выдача заблокирована до
   выполнения `activation_condition`). Парсинг ID всегда `id[:2]` / `id[2:4]` / `id[4:]` —
   никогда фиксированным срезом вроде `id[4:8]`, потому что длина счётчика растёт со временем.

6. **`scripts/id_generator/<entity>_id.py`** (эта директория, например `model_id.py`,
   `run_id.py`) — единственный санкционированный способ для кода `services/*` и
   studbook-writer'ов запросить ID; см. таблицу выше.

7. **`studbook/templates/model_passport_template.yaml`** — что происходит с `model_id` СРАЗУ
   ПОСЛЕ того, как `issue_model_id()` его выдал: единая форма профиля модели (одна и та же для
   LLM и классических ML — неприменимые полю поля оставляются пустыми, а не удаляются).
   `resource_class` включает `classical_ml` отдельным бакетом наравне с LLM-размерами.
   `parent_model_id` в форме уже есть (nullable), хотя `PE`/lineage ещё не активирован. Поля
   `dataset_id`/`training_config_id`/`selector_version`/`eval_score`/`bereiter_report_id`/
   `decision_reason` сюда намеренно **не входят** — это факты `farm.runs`/`farm.samples`/будущих
   таблиц Bereiter или содержимое `studbook/breeding_decisions/`/`worker_admissions/`, а не факты
   самой модели (см. `database_guide.md` §9.1, §4.1 — синхронизирован с этим шаблоном). Реальных
   заполненных профилей в `studbook/model_registry/models/` пока нет — моделей ноль.

## 3. Правила, которые нельзя нарушать (сведены сюда намеренно, из разных файлов выше)

- Не изобретать `full_prefix` инлайн. Незарегистрированный или отложенный домен всегда должен
  упасть с исключением — предложение нового префикса идёт через пользователя, не автоматически.
- Не парсить ID фиксированными срезами — только через `parse_id()`.
- Не писать в `id_counters.json` напрямую (руками, через `open()` без блокировки, или из другого
  модуля) — единственный вход это `core.py`.
- Не путать `GE` и `GN` — см. пункт 4 выше.
- Не активировать отложенный домен "на всякий случай" — только вместе с реализацией
  соответствующей таблицы/сервиса, который его реально использует (см. `ID_DOMAINS.md`, раздел
  "Правило на будущее").
